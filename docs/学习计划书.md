## 一、项目质量评估

### 1.1 整体评价：**7.5/10 — 值得深入学习**

| 维度 | 评分 | 说明 |
|------|:--:|------|
| 架构设计 | 8/10 | 前后端分离清晰，分层合理（controller → service → model），RAG 管道设计专业 |
| 代码质量 | 7/10 | TypeScript 全栈类型安全，但也有巨型组件（DocumentReader.vue ~840 行）、少量死代码 |
| 技术选型 | 8/10 | Vue 3 + Express 5 + LanceDB + LangChain，技术栈现代且实用 |
| 文档完善度 | 8/10 | CONTEXT.md 很详细，有 API 文档、部署指南、改造计划书 |
| 工程化 | 7/10 | Docker 部署、90 个测试用例、ESLint；但缺 CI/CD、用 sequelize.sync() 而非迁移 |
| 安全性 | 7/10 | JWT httpOnly、RBAC、Zod 校验、限流；但改造计划书自曝了鉴权漏洞 |
| AI 能力 | 9/10 | 双路检索 + RRF 融合 + Rerank + MMR，Text2SQL，文档分类，题目提取/生成 |

### 1.2 亮点

- **完整的 RAG 管道**：从文档上传 → 语义切块 → 向量化 → 双路检索 → RRF 融合 → Rerank 精排 → MMR 去重 → LLM 生成的完整链路，市面上很少有教学项目做到这个深度
- **学术级的检索架构**：向量相似度 + BM25 全文索引的混合检索，加上 Reciprocal Rank Fusion 融合和 Maximal Marginal Relevance 去重，这些都是工业级 RAG 系统的标配
- **题库学习闭环**：不只是文档管理，而是"上传 → 解析 → 出题 → 练习 → 自评"的完整学习闭环

### 1.3 不足（不影响学习价值）

- 部分组件偏大（DocumentReader.vue ~840 行），可作为"如何拆分组件"的练习素材
- 用 sequelize.sync() 管理表结构（生产环境不推荐），但作者已在改造计划书中规划迁移到 umzug
- 缺少 CI/CD 流程、E2E 测试

### 阶段 3：RAG 深度学习（第 15-18 天，约 10 小时）

**目标**：这是本项目最有价值的部分，值得花整块时间深入理解。

#### 第 15 天 — Embedding 与向量检索
- 理解什么是 Embedding：文字 → 数字向量的过程
- 理解余弦相似度：两个向量怎么比"远近"
- 手动写一个简单的向量检索 Demo（用纯 JS 的余弦相似度函数，不依赖库）
- 阅读 [backend/src/services/embedding.service.ts](backend/src/services/embedding.service.ts)

#### 第 16 天 — 混合检索 + RRF
- 理解 BM25 算法原理（词频、逆文档频率、文档长度归一化）
- 理解为什么"向量检索 + BM25"比单独一个更好：
  - 向量擅长语义相近（"汽车"能匹配到"轿车"）
  - BM25 擅长关键词精确匹配（"2024年财报"）
- 手动实现 RRF 融合算法：
  ```
  score = SUM( 1 / (k + rank_i) )  for each rank list i
  ```
- 阅读 [backend/src/services/retrieval.service.ts](backend/src/services/retrieval.service.ts) 中的 RRF 实现

#### 第 17 天 — Rerank + MMR
- 理解 Rerank 的原理：双塔模型 vs 交叉编码器（Cross-Encoder）
  - 向量检索用的是"双塔"（快但粗糙），Rerank 用的是"交叉编码器"（慢但准）
- 理解 MMR（Maximal Marginal Relevance）：
  - 公式：`MMR = λ·相似度(query, doc) - (1-λ)·max(相似度(doc, 已选中的doc))`
  - 目的：让检索结果多样化，不要返回 5 个意思差不多的片段

#### 第 18 天 — RAG 全链路复盘
- 手画完整的 RAG 流程图（用户提问 → 最终回答的每一步）
- 写出每一步的输入和输出
- 思考：如果让你改进这个 RAG 系统，你会从哪里入手？

**产出**：一份 RAG 学习笔记（Embedding、混合检索、RRF、Rerank、MMR 的原理和代码对应）。

## 四、学习技巧

### 4.1 代码阅读顺序

不要试图一行行读完所有代码。按下面的优先级：

1. **入口文件** → 了解启动流程
2. **路由文件** → 了解有哪些功能
3. **核心页面/服务** → 理解核心业务
4. **工具/辅助** → 用到再看

### 4.2 使用 Claude Code 辅助

在项目中使用以下命令：

```
/claude-code-guide     # 询问 Claude Code 的用法
/debug                 # 调试时使用
/review                # 写完代码后请 AI 审查
/test                  # 让 AI 帮你写测试
```

遇到不理解的代码，直接问 AI："这段代码在做什么？为什么这样写？"

### 4.3 记录学习笔记

建议每天记录：
- 今天读了哪些文件
- 理解了哪个概念
- 哪个地方没看懂（第二天优先解决）

## 六、推荐延伸学习

完成本项目学习后，可以顺着以下方向深化：

1. **向量数据库**：对比 LanceDB、Chroma、Qdrant、Milvus 的优劣
2. **LangChain 框架**：本项目用到了部分 LangChain，可以系统学习 LangChain 的 Chain/Agent/Tool 体系
3. **大模型应用开发**：Prompt Engineering、Function Calling、Agent 架构
4. **前端进阶**：Vue 3 的 Suspense、Teleport、自定义指令
5. **后端进阶**：微服务拆分、消息队列（文档处理异步化）、分布式部署
