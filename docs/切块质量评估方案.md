# 切块质量评估方案

> 原则：切块时一次性判断质量，不靠检索后补救。零 API 调用，纯规则。

---

## 为什么需要

检索后的覆盖度闸门和 chunk 展开本质是在补偿切块阶段的缺陷。把质量判断前移到切块环节，一次计算、永久复用。

## 三个信号

### 信号 1：边界完整性 `boundaryOk`

```ts
function checkBoundary(chunk: string): boolean {
  if (chunk.length < 10) return false
  if (/^[，。；、）\)\]】]/.test(chunk)) return false  // 开头是标点 = 从句子中间切开
  return /[。！？\.!\?]$/.test(chunk)                 // 结尾是完整标点 = 没被截断
}
```

目标识别：
- `ef、forwardRef等API` — 开头无主语，从句子中间切开 → `boundaryOk=false`
- `React 使用虚拟 DOM 来提升渲染性能。` — 开头主语清晰、结尾标点完整 → `boundaryOk=true`

### 信号 2：信息密度 `infoDensity`

```ts
function evaluateDensity(chunk: string): number {
  const navNoise = ['目录', '前言', '点击', '下一页', '上一页', '返回', '版权', '免责']
  const hits = navNoise.filter(w => chunk.includes(w)).length
  const ratio = 1 - hits / navNoise.length
  const fragPenalty = chunk.length < 50 ? 0.3 : 0   // 碎片扣分
  return Math.max(0, ratio - fragPenalty)
}
```

目标识别：
- 目录页/导航页被切成 chunk → `infoDensity < 0.5`
- < 50 字的微型 chunk → `infoDensity < 0.7`
- 正常内容段落 → `infoDensity ≈ 1.0`

### 信号 3：主题锚定 `anchorHeadings`

```ts
function anchorHeadings(chunks: ChunkResult[]): void {
  let currentHeading: string | null = null
  for (const c of chunks) {
    if (c.heading) { currentHeading = c.heading; continue }
    if (currentHeading) c.heading = currentHeading
  }
}
```

同一个章节下的连续 chunk 共享 heading。检索时 LLM 看到的不是孤立片段，而是带了章节标签的上下文。

## 执行时机

`splitIntoChunks()` 返回前，`addOverlap()` 执行后：

```
全文 → semanticChunking / paragraphChunking → addOverlap → assessQuality → 返回
```

一次上传只算一次，不重复。

## 使用方式

### 日志输出

上传文档后会看到：

```
[chunking] 边界质量问题: 3 个 chunk 切割不完整, index=[4,12,31]
[chunking] 信息密度问题: 2 个 chunk 含导航词/碎片, index=[0,44]
```

### 检索排序（可选扩展）

```ts
// retrieval.service.ts — MMR 选择时微调 score
adjustedScore = chunk.score * (1 + (chunk.boundaryOk ? 0.05 : -0.05) + (chunk.infoDensity - 0.7) * 0.1)
```

同分数下优先选完整、密度高的 chunk。当前未实施，作为后续可选优化。

## 和检索后补救的对比

| | 检索后补救（覆盖度闸门） | 切块质量评估 |
|---|---|---|
| 时机 | 每次查询 | 上传时一次 |
| 成本 | 每次检索额外逻辑 | 零 |
| 能否排序 | 不能，只做拦截 | 能，质量分参与排序 |
| 解决什么问题 | 拦掉坏 chunk（治标） | 标记坏 chunk（治本） |
| 数据库存储 | 不需要 | boundaryOk + infoDensity 两个字段 |

## 测试方法

上传一个文档，看日志是否输出质量信息：

```
docker logs ai-web-backend-1 | grep "边界质量\|信息密度"
```

期望有输出，说明质量评估已生效。
