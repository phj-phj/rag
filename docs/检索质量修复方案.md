# 检索质量修复方案

> 基于生成评估暴露的两个核心问题：
> 1. 忠实度 — 召回的 chunk 内容不足以支撑回答时，LLM 被迫编造（q06 翻车）
> 2. 完整性 — chunk 太小截断了关键上下文，答案遗漏关键信息点（q08/q10 翻车）

---

## 问题一：忠实度守卫 — chunk 不够时不生成

### 现象

q06 "什么是微服务架构" faithfulness=2，幻觉 4 次。检索召回了 doc 30（大厂后端面试真题），但命中的 chunk 是目录/框架段落，没有微服务的实际讨论内容。LLM 被迫调用内置知识硬编。

当前代码已有低分过滤：

```typescript
const MIN_SCORE = 0.1
const filtered = retrieved.filter(r => r.score >= MIN_SCORE)
```

q06 的 chunk score 在 0.15-0.16，过了 0.1 的线，但内容不相关。**分数阈值区分不了"内容不相关"和"分数低但内容勉强够"。**

### 方案：生成前做一次轻量覆盖度检查

在检索完成后、LLM 生成前，加一个闸门。

```
检索 → 覆盖度检查 → 通过 → LLM 生成
                  → 不通过 → 返回"未找到相关信息"
```

#### 覆盖度怎么判断

三个信号一起用，不依赖单一指标：

```typescript
async function checkCoverage(
  question: string,
  chunks: { content: string; score: number }[],
): Promise<{ sufficient: boolean; reason: string }> {

  // 信号 1：总可用文本量
  const totalText = chunks.reduce((s, c) => s + c.content.length, 0)
  if (totalText < 300) {
    return { sufficient: false, reason: `chunk 总文本量 ${totalText} 字 < 300` }
  }

  // 信号 2：top chunk 的质量
  const topScore = chunks[0]?.score ?? 0
  if (topScore < 0.15) {
    return { sufficient: false, reason: `最高分 ${topScore.toFixed(3)} < 0.15` }
  }

  // 信号 3：关键词命中率（快速、0 API 调用）
  const questionWords = question.replace(/[？?。，,、\s]/g, '').slice(0, 30)
  const combinedText = chunks.map(c => c.content).join(' ')
  let hitCount = 0
  for (const ch of questionWords) {
    if (combinedText.includes(ch)) hitCount++
  }
  const keywordHitRate = hitCount / questionWords.length
  if (keywordHitRate < 0.3 && topScore < 0.2) {
    return { sufficient: false, reason: `关键词命中率 ${(keywordHitRate * 100).toFixed(0)}% 过低` }
  }

  return { sufficient: true, reason: 'ok' }
}
```

三个信号的设计逻辑：
- **信号 1**：文本太少一定不够
- **信号 2**：最高分太低说明最相关的 chunk 也不可信
- **信号 3**：关键词几乎不命中说明 chunk 跟问题语言上不重合（语义也不大可能重合）

#### 修改 chat.controller.ts

```typescript
// 检索后、生成前
const MIN_SCORE = 0.1
const filtered = retrieved.filter(r => r.score >= MIN_SCORE)

if (filtered.length === 0) {
  // 现有逻辑：无相关片段
  return res.json({ answer: '当前文档库中未找到相关信息。', model: 'retrieval', docs: [] })
}

// 新增：覆盖度闸门
const coverage = await checkCoverage(question, filtered.map(r => ({
  content: r.content,
  score: r.score,
})))
if (!coverage.sufficient) {
  logger.info(`[chat] 覆盖度不足: ${coverage.reason}`)
  return res.json({ answer: '当前文档库中未找到相关信息，请尝试更换关键词或上传相关文档。', model: 'retrieval', docs: [] })
}
```

#### 效果预测

q06 "什么是微服务架构" 经过覆盖度检查：
- 召回的 chunk 全是面试题目录框架词，关键词"微服务架构优缺点"在这些 chunk 里命中率极低
- 信号 3 的 keywordHitRate < 0.3，topScore 约 0.15
- → 闸门关闭，返回"未找到相关信息"
- → 相比之前 LLM 编造 4 次幻觉，这是一个更诚实的回应

## 问题二：完整性 — chunk 太小，允许展开上下文

### 现象

q08 "Linux 中如何查找某个文件" 完整性=1。检索找到了正确文档（doc 32），但返回的 chunk 只是一小段文本，缺少具体的 `find` 命令格式、`grep` 参数、`chmod` 示例。这些内容存在于相邻 chunk 中，但被切块边界切断了。

q10 "Go 语言中怎么使用 Redis" 同病：chunk 里是目录框架词，没有 `go-redis` 的代码示例。

### 根因

语义分块按相似度断点切分后，相邻 chunk 之间的内容有逻辑上的连续性。当前检索只返回 top-K 个独立的 chunk，丢弃了它们周围的上下文。

### 方案：检索结果自动展开邻近 chunk

#### 第零步：确认数据基础

MySQL 的 `Document_Chunks` 表已有 `chunk_index` 字段：

```sql
Document_Chunks: id, document_id, chunk_index, content, token_count, strategy, heading
```

不需要重新索引 LanceDB，直接用 MySQL 查相邻 chunk。

#### 展开逻辑

```typescript
// retrieval.service.ts 新增

async function expandNeighbors(
  chunks: RetrievedChunk[],
  expandCount: number = 1,  // 每侧展开 N 个邻居
): Promise<RetrievedChunk[]> {
  // 只为 top 3 展开（第 4-5 名通常分数已经很低，展开意义不大）
  const toExpand = chunks.slice(0, 3)

  // 批量查询 MySQL Document_Chunks 获取 chunk_index
  const chunkIds = toExpand.map(c => c.chunkId)
  const rows = await DocumentChunk.findAll({
    where: { id: chunkIds as number[] },
    attributes: ['id', 'document_id', 'chunk_index'],
  })

  const indexMap = new Map<number, { docId: number; chunkIdx: number }>()
  for (const r of rows) {
    indexMap.set(r.id, { docId: r.document_id, chunkIdx: r.chunk_index })
  }

  // 收集需要取的邻居 chunk_id
  const neighborChunkIds = new Set<number>()
  for (const c of toExpand) {
    const info = indexMap.get(c.chunkId)
    if (!info) continue
    for (let offset = -expandCount; offset <= expandCount; offset++) {
      if (offset === 0) continue
      neighborChunkIds.add(info.chunkIdx + offset) // 这里存的是 chunk_index，需要进一步查
    }
  }

  // 查 MySQL 获取邻居 chunk
  // 对每个 (docId, chunkIdx ± offset) 查询
  const neighborMap = new Map<number, string>() // chunkId → content
  for (const c of toExpand) {
    const info = indexMap.get(c.chunkId)
    if (!info) continue
    const neighbors = await DocumentChunk.findAll({
      where: {
        document_id: info.docId,
        chunk_index: [
          info.chunkIdx - expandCount,
          info.chunkIdx - 1,
          info.chunkIdx + 1,
          info.chunkIdx + expandCount,
        ].filter(n => n >= 0 && n !== info.chunkIdx),
      },
      attributes: ['id', 'content', 'chunk_index'],
    })
    for (const n of neighbors) {
      neighborMap.set(n.id, n.content)
    }
  }

  // 将邻居内容拼接到原 chunk
  return chunks.map(c => {
    const neighborKeys = [...neighborMap.keys()] // 简化示意
    // 实际实现：把邻居 content 拼到原 chunk 后面
    // ...
    return c
  })
}
```

#### 简化版方案（推荐先做）

上面的方案涉及多次 MySQL 查询。可以先做一个更简单的版本：

**当 chunk 文本 < 200 字时，直接查询同文档的相邻 chunk，不判断 chunk_index：**

```typescript
async function expandContext(
  chunks: RetrievedChunk[],
): Promise<RetrievedChunk[]> {
  const MIN_CHUNK_SIZE = 200

  for (const c of chunks.slice(0, 3)) {
    if (c.content.length >= MIN_CHUNK_SIZE) continue

    // chunk 太小，从 MySQL 查同文档的前后各一条
    const neighbors = await DocumentChunk.findAll({
      where: { document_id: c.documentId },
      order: [['chunk_index', 'ASC']],
      attributes: ['id', 'content', 'chunk_index'],
    })

    // 找到当前 chunk 的位置，取前后各一条
    const idx = neighbors.findIndex(n => n.id === c.chunkId)
    if (idx < 0) continue

    const parts: string[] = []
    if (idx > 0) parts.push(neighbors[idx - 1].content)
    parts.push(c.content)
    if (idx < neighbors.length - 1) parts.push(neighbors[idx + 1].content)

    c.content = parts.join('\n')
  }

  return chunks
}
```

#### 在 retrieve() 中集成

```typescript
// retrieve() 内部，MMR 之后、return 之前
if (opts?.expandContext !== false) {
  selected = await expandContext(selected)
  logger.info(`[retrieval] 上下文展开完成`)
}
```

### 什么时候展开

| 条件 | 动作 |
|------|------|
| chunk 内容 ≥ 200 字 | 不展开，内容已足够 |
| chunk 内容 < 200 字 | 展开 ±1 邻居 |
| chunk 来自较长文档 (>50 chunks) | 展开 ±2 邻居（隔离风险高） |

## 两个方案的交叉效果

```
q06 "微服务架构" → 覆盖度检查拦截 → 不调用 LLM → faithfulness 问题消失
q08 "Linux 文件查找" → chunk 从小变完整 → LLM 拿到足够信息 → completeness 回升
q10 "Go 用 Redis" → chunk 从小变完整 → LLM 能输出代码示例 → completeness 回升
```
