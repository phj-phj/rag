#### 项目简介

**原文**：
> Papier 是一款基于 RAG 技术的智能学习平台，支持 PDF/Word/TXT 文档上传、自动解析、AI 问答和题目生成，帮助用户将学习材料转化为考试题目。前后端分离架构，支持 Docker 一键部署。

**问题**：
- 无痛点：没回答"为什么需要这个产品"——人工出题有什么问题？
- 无结果：缺失 RedologFiles 中记录的核心量化指标
- 无技术深度：读起来像课程设计项目介绍

**改后版本**：

> **Papier — 端到端 RAG 智能学习平台**（独立开发）
>
> 针对传统备考中**人工出题耗时、题目质量不稳定、AI 幻觉导致答案不可信**三大痛点，
> 从零设计并实现了完整的多路 RAG 问答与题目自动生成系统。
> 核心包含双路检索增强问答引擎、三层防线的 AI 题目生成管道、智能每日训练模块。
> **将 AI 幻觉率从 56% 压降至 11%，回答忠实度从 4.1/5.0 提升至 4.9/5.0，
> 预生成题目忠实度达 4.7/5（闸门拒收率 <40%）**。
> 支持 DeepSeek / 智谱 / SiliconFlow 多模型后端，月均 AI 成本控制在 5-10 元。
> 已部署至阿里云 ECS 生产环境，编写 90 条自动化测试用例，Swagger 文档覆盖 39 个 API 端点。

#### 技术栈

**原文**：
> Vue 3、TypeScript、Vite、Tailwind CSS、Pinia、Three.js、ECharts、Express、Sequelize、MySQL、LanceDB、DeepSeek、LangChain、智谱 Embedding、SiliconFlow Reranker、Docker、Nginx

**问题**：
- 18 个名词无分类无用途说明，面试官扫一眼就过
- RedologFiles 中记录的很多技术细节（pdfjs-dist、Winston、Zod、JWT httpOnly Cookie）被漏掉，但堆在列表里也不对——需要分层说明

**改后版本**：

```
- 前端渲染层：Vue 3 + TypeScript + Vite + Tailwind CSS + Pinia
              PDF.js (Canvas 逐页渲染，3.x 兼容移动端)、ECharts 可视化、Three.js 3D 图表
- 后端服务层：Express 5 + TypeScript + Sequelize ORM + Zod 请求校验
              Winston 结构化日志（22 文件统一）、Swagger API 文档（39 端点）
- 大模型层：DeepSeek-v4-flash（问答/出题）、DeepSeek-Reasoner（复杂推理）
            阿里云百炼 / DeepSeek 原生 API 双后端可切换
- AI 框架层：LangChain（LLM 调用链抽象），全链路 OpenAI 兼容协议，模型零成本替换
- 检索增强层：LanceDB 向量库（cosine 1024 维）+ BM25 全文检索双路并行召回
              RRF 动态加权融合 + Cross-Encoder 重排序 + MMR 多样性选择
- 质量保障层：智谱 embedding-2（文本向量化）、SiliconFlow Qwen3-Reranker-0.6B（精排）
              独立 LLM 忠实度审查闸门、覆盖率闸门
- 工程化：Docker Compose 三服务编排 + Nginx 反向代理（SSE 流式关闭缓冲）
          JWT httpOnly Cookie（localStorage 全量迁移）、express-rate-limit
- 存储层：MySQL 8.0（9 表关系模型）+ LanceDB（向量 + BTREE + FTS 三索引）
```

#### 个人职责（第 1 条）— 多路 RAG 检索系统

**原文**：
> 实现了基于 LanceDB 的 RAG 检索链路，包括文档切块、向量化存储、语义检索与 BM25 关键词检索，使用 RRF 融合多路召回结果，并通过 Reranker 精排提升检索精度。

**问题**：
- 只描述动作，没说为什么需要双路检索、为什么需要 RRF 融合
- 缺失 RedologFiles 中记录的量化结果（幻觉率 56%→11%、检索区分度 1.3×→497×）
- "实现了"关键词——暗示这是照着文档做的，不是自己设计的

**改后版本**：

> **主导设计多路 RAG 检索系统（核心贡献）**
>
> 针对单路向量检索在高维语义与关键词精确匹配间无法兼顾、检索噪声导致 LLM 幻觉频发的问题（初始幻觉率 56%），
> 设计并实现了五层检索管线：
> ① 双路并行召回（向量 cosine + BM25 全文检索，LanceDB 三索引）
> ② RRF 动态加权融合（自研算法：基于各路径信号密度自动分配权重，FTS 弱时其权重自动压至 0.11，FTS 不可用时归零）
> ③ 邻块上下文扩展（短 chunk < 200 字拼接同文档前后邻居，补全语义后再进入精排）
> ④ Cross-Encoder 重排序（SiliconFlow Qwen3-Reranker-0.6B，检索区分度从 1.3× 提升至 497×）
> ⑤ MMR 多样性选择（Jaccard 重叠惩罚，λ=0.7）+ 覆盖率闸门（总文本 < 300 chars 拦截；top1 < 0.10 绝对底线拦截；灰色区间 [0.10, 0.20) 内 PeakGap < 0.15 拦截，防低分扎堆误放行）
>
> 最终将 **AI 幻觉率从 56% 降至 11%（-45pp），回答忠实度从 4.1/5.0 提升至 4.9/5.0**
> 消融实验量化了每个组件的贡献：双路检索 Recall@5=100%（仅向量 95%），
> 动态加权使 MRR 从 0.7583 提升至 0.8750（+15.4%）。

#### 个人职责（第 2 条）— AI 题目生成管道

**原文**：
> 基于 LLM 实现了题目自动生成功能，支持从知识文档预生成题目和从题库文档提取已有题目，使用 Prompt Engineering 优化生成质量，并添加忠实度验证环节。

**问题**：
- "基于 LLM 实现了"——像调了个 API
- 完全没有体现 RedologFiles 中记录的三层防线设计
- "添加忠实度验证环节"——最核心的质量闸门一笔带过
- 缺失提取 Prompt 三次迭代的工程细节（v1→v3.1 的口语过滤、指代消解、独立可读性）

**改后版本**：

> **设计 AI 题目生成管道（三层质量防线）**
>
> 针对 LLM 生成内容存在事实性错误、题库文档提取质量参差的问题，设计了三层递进防线：
> ① **生成前**：语义密度过滤器 — 10 个正则模式检测实体（公式/数字/专有名词/代码标识符等），
>    密度 < 1 实体/100 字符的叙述性段落直接跳过，从源头减少 LLM 对无知识点文本的"硬编"
> ② **生成中**：受限 Prompt 工程（告知模型"你只能看到这份文档，没有学过任何编程语言"，禁止使用外部知识）
> ③ **生成后**：独立 LLM 忠实度审查闸门 — 使用 deepseek-reasoner（与生成模型异构，避免自评盲区），
>    `temperature=0`，逐断言对照原文，faithfulness < 4/5 拒收；
>    连续拒收 ≥5 题时自动触发回退——对剩余原文做更细粒度的重新分块（3000→1500 字/块），
>    小块上下文约束更强，LLM 更贴原文
>
> 支持三种来源：题库文档智能提取（extracted）、知识文档预生成（ai_pregenerated）、
> 训练时动态生成（ai_adhoc）。提取 Prompt 经历 v1→v3.1 三次迭代：
> 新增口语注释过滤、指代词消解（"这个""那种"→通顺主语）、独立可读性自检规则，
> 不合格题目（口语注释/无对象/无问号/截断）从 22 道/文档降至 0。
> 预生成忠实度达 **4.7/5**。

#### 个人职责（第 3 条）— AI 智能路由与多轮对话

**原文**：
> 实现了 AI 问答的智能路由功能，根据问题类型自动选择 Text2SQL、RAG 检索或直接回答三种策略。支持多轮对话记忆和历史相关性筛选。

**问题**：
- 又是"实现了"——没有设计决策的思考过程
- Text2SQL 的安全设计（SQL 注入防护、密码字段屏蔽）完全没提——这恰恰是面试官最想听的工程化细节
- "历史相关性筛选"的巧妙设计（一次 LLM 调用同时做改写+筛选）被省略了

**改后版本**：

> **设计 AI 智能路由与多轮对话系统**
>
> 针对用户问题类型多样（数据统计 / 文档问答 / 元问题闲聊）、单一检索策略无法通配的问题，
> 设计了两级路由决策：规则层快速分流（24 个特征词匹配，<1ms）+ LLM 层精准三路分类
> （SQL → Text2SQL 执行 / DOCS → RAG 检索 / 直接回答），SQL 路径优先避免被 RAG 误判。
>
> Text2SQL 模块实现全链路安全管控：8 表 Schema 注入 Prompt + SELECT-only 白名单 +
> DML/DCL 拦截 + 多语句阻断 + 自动 LIMIT 100 + 运行时 Password 字段强制删除。
>
> 多轮对话方面，设计"查询改写 + 历史过滤"**单次 LLM 调用合并方案**：
> 用一次 LLM 调用同时完成追问关键词提取和历史 Q&A 相关性判定，
> 在控制 Token 消耗的前提下维持多轮对话上下文连贯性。
> 流式 SSE 三路统一（SQL/direct/RAG 均走 `res.write`），前端不再出现 JSON 解析失败。

#### 个人职责（第 4 条）— 文档处理管道

**原文**：
> 实现了 PDF/Word/TXT 文档的上传解析功能，包括文本提取、HTML 转换、页眉页脚过滤、语义分块，并在前端使用 pdfjs-dist 渲染 PDF。

**问题**：
- 完全没体现 RedologFiles 中记录的实战复杂度——pdfjs-dist 6.x→3.x 降级移动端兼容、中文文件名编码修复、MIME 类型 Bug
- 语义分块的核心算法（句子 → embedding → 余弦相似度断点）被简化成了"语义分块"四个字
- 题库文档的特殊处理（题号检测、强制按题号分块）没提

**改后版本**：

> **构建文档处理全链路**
>
> 后端实现多格式文本提取管道：PDF（pdf-parse + unpdf 双引擎，支持内嵌图片 base64 提取）、
> DOCX（JSZip + XML 解析 + 内嵌图片提取）、TXT。包含页眉/页脚自动过滤（跨页重复行检测，60% 阈值）、
> 页码模式识别（纯数字 / "1/100" / "第1页" / "- 1 -"）、HTML 结构化转换（段落/标题/列表/代码块）。
> 修复中文文件名上传乱码（`Content-Disposition` UTF-8 字节 latin1 误解析 → `Buffer.from` 转码）、
> Word MIME 子类型错误解析（`application/msword` → `"doc"` 而非 `"msword"`）。
>
> 语义分块算法：句子分割 → embedding 向量化 → 相邻句子余弦相似度 → 低于阈值 0.5 切断 →
> min/max 块大小合并控制（150-500 字），重叠窗口 20%。题库文档特殊适配：
> 题号模式检测（`1.` / `(2)` / `一、`），≥3 个题号强制按题边界分段，不拆散 Q&A 对。
>
> 前端 PDF 渲染经历 pdfjs-dist 6.x → 3.x 降级迁移：6.x ESM Worker（`.mjs`）在移动端
> （iOS Safari / Android WebView）完全不兼容 → 降级至 3.x 传统 Worker（`.js`），
> 保持 Canvas 逐页渲染 + devicePixelRatio 适配 + 缩放工具栏，bundle 缩减 29%（479KB → 339KB）。

#### 个人职责（第 5 条）— 工程化与安全

**原文**：
> 负责项目部署，使用 Docker Compose 和 Nginx 部署到阿里云 ECS，编写了单元测试和集成测试，并配置了 Swagger API 文档。

**问题**：
- 最严重的问题——RedologFiles 中大量工程化细节全部消失：
  - JWT localStorage → httpOnly Cookie 安全迁移（13 处代码删除）
  - Winston 日志统一（22 文件，100+ console.log → logger）
  - 嵌入式向量数据库选型与启动修复（并发事务冲突）
  - 限流器信任代理修复、Embedding 429 重试
  - 部署踩坑 10+ 个问题的排查能力完全没体现

**改后版本**：

> **全链路工程化落地与安全加固**
>
> **安全**：完成 JWT 存储方案升级——从 localStorage（XSS 可窃取）迁移至 httpOnly Cookie，
> 后端 5 文件（cookie-parser 集成 + `sameSite: 'lax'` + HTTPS 自适应 `secure`）+ 前端 5 文件
> （删除 13 处 localStorage 操作 + `withCredentials: true`），向后兼容 Authorization header。
> Text2SQL 密码安全：Prompt 禁止查询 password + 运行时强制 strip。
>
> **测试体系**：从零搭建 Vitest 测试框架，编写 **90 条用例**（74 单元 + 16 集成）。
> 单元覆盖：余弦相似度计算、RRF 融合、MMR 去重、Jaccard 重叠、流式 JSON 增量解析状态机、
> 题目提取正则、分块死循环守卫。集成覆盖：auth 全流程 + 文档 CRUD + 权限校验。
> 抓到真实 bug：`splitForExtraction(text, 0)` 死循环至数组溢出。
>
> **可观测性**：统一 22 个文件的日志系统——100+ `console.log` → Winston 结构化日志 +
> 模块标签 + DEBUG_AI 开关 + 文件持久化 + 滚动覆盖。调试透明化：AI 全链路每步骤耗时与结果日志。
>
> **部署**：Docker Compose 三服务编排 + Nginx 反向代理（SSE `proxy_buffering off`）+
> PM2 进程管理。阿里云 ECS 2核2G 生产环境部署，经历 10+ 采坑修复：
> Node 版本升级（18→20）、swap 解决前端 OOM、httpOnly Cookie HTTP/HTTPS 适配、
> LanceDB 启动并发冲突、Embedding 429 限流重试、AI 模型切换（3 次更换后端）。
> Swagger 文档覆盖全部 39 个 API 端点（8 组）。

## 核心方法论

**三段式改写公式**：`我解决了什么问题` → `我用了什么方案` → `我取得了什么成果`

**审查维度**：痛点驱动 → 架构设计 → 量化结果 → 工程化能力
