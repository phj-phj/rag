# Papier 题库系统改造计划书

> 版本：v1.0 | 日期：2026-06-01 | 状态：待实施

---

## 一、项目概述

### 1.1 背景

Papier 当前定位为"团队文档管理与分享平台"，具备文档上传、多格式预览、RAG 智能问答等能力。现需从通用文档管理向**题库学习平台**转型，核心场景为：用户上传题库文档或知识资料，系统自动提取/生成论述题，用户以翻卡片方式自评学习。

### 1.2 改造目标

1. **数据结构化**：新建 Question 表，将"题目"从文档文本块中独立出来
2. **AI 智能出题**：上传文档后自动检测类型（题库/知识），异步提取或生成题目入库
3. **学习交互**：训练页面支持按知识点+难度筛选，翻卡片式浏览，用户自评"已掌握/需复习"
4. **安全规范**：统一输入校验、修复鉴权漏洞、添加速率限制
5. **工程质量**：引入全局错误处理、结构化日志、数据库迁移、核心测试

### 1.3 不改动范围

- JWT 密钥机制（暂不修改）
- 现有文档上传/预览/RAG 问答功能
- 前端 warm-tone 设计主题 + Tailwind v4
- Nginx + pm2 部署架构
- 本地文件存储 + LanceDB 向量库

## 三、改造方案

### 3.1 新增数据模型

#### Question 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| stem | TEXT NOT NULL | 题干 |
| explanation | TEXT NOT NULL | 详细解析/参考答案（合并） |
| type | ENUM('essay') | 题型（当前仅论述题，预留扩展） |
| difficulty | TINYINT | 难度中位数 1-5（初始 NULL） |
| difficulty_votes | JSON | 用户投票记录，上限 20 个 |
| knowledge_point | VARCHAR(200) | 知识点（AI 预标注） |
| source_type | ENUM | 来源：extracted / ai_pregenerated / ai_adhoc |
| source_document_id | INTEGER | 关联文档 ID |
| created_at / updated_at | DATETIME | 时间戳 |

#### PracticeRecord 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER PK | 主键 |
| user_id | INTEGER NOT NULL | 用户 ID |
| question_id | INTEGER NOT NULL | 题目 ID |
| status | ENUM('mastered', 'review') | 已掌握 / 需复习 |
| created_at | DATETIME | 标记时间 |

### 3.2 AI 文档类型检测

上传文档时，基于**段落间语义连贯性**自动判定类型：

- **题库文档**（论述题集）：段间相似度方差大，题与题之间内容跳跃
- **知识文档**（总结材料）：段间相似度平稳，连续论述，过渡平滑

### 3.3 双轨处理流程

- **题库文档**：全文提取 → LLM 识别 Q&A 对 → 写入 Question 表（source_type='extracted'）；同时走 RAG 分块保留检索能力
- **知识文档**：正常 RAG 分块 → 按 1:100 token 比例预生成题目 → 写入 Question 表（source_type='ai_pregenerated'）

### 3.4 训练时智能出题

用户发起训练请求 → 先查 Question 表匹配数 → 题库足（≥2N）则 80% 抽取，题库一般（0.5N~2N）则 60% 抽取，题库不足则全走 AI 生成。指定难度时纯抽取不走生成。

### 3.5 训练页面交互

保持当前全展开卡片布局，新增：
- 知识点 + 难度 + 数量三个筛选条件
- 已收录题和 AI 生成题分区展示，样式统一
- 已收录题底部显示难度投票（5 星）
- 每题底部"已掌握 / 需复习"两个按钮
- 难度投票实时取中位数，满 20 票锁定

### 3.6 安全加固

- **Zod 统一校验**：所有 POST/PUT/DELETE 接口注入
- **鉴权修复**：`/api/admin/stats` 移到 requireAdmin 之后
- **速率限制**：登录 5次/分钟，注册 3次/分钟

### 3.7 工程质量

- **全局错误处理**：AppError 类体系 + errorHandler 中间件 + 脱敏规则
- **结构化日志**：Winston + 请求追踪 ID（x-request-id）
- **健康检查**：GET /health 端点
- **数据库迁移**：umzug 替代 sequelize.sync()
- **测试**：Vitest + supertest，核心链路覆盖
- **组件拆分**：单文件 ≤500 行，提取 composables
- **死代码清理**：删除 6 个遗留文件

### 3.8 新增 API

| Method | Path | 说明 |
|--------|------|------|
| POST | `/api/training/generate` | 混合出题（抽取+生成） |
| POST | `/api/training/generate/stream` | SSE 流式出题 |
| POST | `/api/training/record` | 标记已掌握/需复习 |
| POST | `/api/training/difficulty` | 难度投票 |
| GET | `/api/training/review` | 获取复习列表 |
| GET | `/api/questions` | 题目查询 |

## 七、验收标准

1. 上传知识文档后，Question 表有对应预生成题目（数量 ≈ 文档 token / 100）
2. 上传题库文档后，题目被正确提取入库
3. 训练页面可按知识点 + 难度筛选，已收录题和生成题分区展示
4. 难度投票功能可用，中位数实时生效
5. 标记"需复习"后，复习页面可筛选出对应题目
6. 所有 POST/PUT/DELETE 接口有 Zod 校验
7. `/api/admin/*` 所有接口需要 admin 权限
8. 登录接口超过 5 次/分钟返回 429
9. `GET /health` 返回数据库连接状态
10. 全局错误不泄露内部信息



1. FTS 是负优化 — MRR 跌 13.3%


仅向量 MRR = 0.875  →  加 FTS MRR = 0.758
这不是配置问题，是信号质量问题。中文 BM25 在面试题 PDF 上的词汇匹配精度不足。文档里充斥着"面试题""Linux 命令""Go 面试"这类目录词，FTS 匹配到这些词就往上推，把真正语义相关的 chunk 挤下去了。q08 和 q09 最明显：仅向量 MRR=1.000，FTS 一开变成 0.500。

这说明他"双路等权 RRF 融合"的设计在这个场景下是错误的。 向量路和 FTS 路应该按数据特征动态加权，而不是永远 1:1。

2. 评估数据集没有区分度 — 100% 说明不了任何问题

10 个 query 全是 easy：问题直接来自文档章节标题，每个问题只标注了一个相关文档。Recall@5 和 Hit@5 在数学上等价。打一个极端比方：用 Ctrl+F 测 10 个题全找到了你也拿不到有意义的结论。

缺少 medium（语义改写）、hard（跨文档综合）、负例（库中没有答案）。

3. MMR 的 Jaccard 度量在中文下基本失效


const setA = new Set(a.slice(0, 200))  // 字符级集合
"微服务架构的优缺点" 和 "微服务架构的通信方式" — 这两个 chunk 讨论同一主题的不同方面，但字符级 Jaccard 重叠度高，MMR 会因为"太相似"把第二个扔掉。而 "微服务架构的优缺点" 和 "Linux 文件查找命令" — 完全不同主题，字符级重叠度低，MMR 会认为它们"很有多样性"而保留。MMR 的多样性定义和真实的语义多样性是脱节的。

这次评测里 MMR 碰巧帮了 2 个 query，是因为 FTS 引入了大量噪声，MMR 的同文档惩罚阴差阳错清了场。换一组数据这个效果会消失。

4. 所有参数仍是拍脑袋定的

参数	当前值	依据
MMR λ	0.7	无
RRF 向量/FTS 权重	1:1	无
RRF K 值	60	业内常见值，但未经验证
语义分块阈值	0.5	无
文档分类阈值	0.08	无
